## Design System

### Atomic Design Pattern

The frontend follows **Atomic Design** principles (v1.2.56+):

**Atoms** - Basic building blocks:
- `Button`: Primary/secondary variants, loading states
- `Icon`: SVG icon wrapper
- `Input`: Form inputs with validation styling
- `Badge`: Status indicators (difficulty, score)

**Molecules** - Simple component combinations:
- `Card`: Content containers with consistent styling
- `FormField`: Label + Input + Error message
- `ListItem`: Reusable list item with icon/actions

**Organisms** - Complex UI sections:
- `Header`: Navigation bar with auth state
- `Modal`: Overlay dialogs
- `ScreenLayout`: Full-screen page wrapper with header/content/footer

**Templates** - Page-level layouts:
- `GameRoundScaffold`: Universal game layout (v1.2.37)
- `RoundActionTray`: Shared action buttons for games (v1.2.38)

## Routing Architecture

The app uses React Router v6 with protected routes:

```tsx
// From src/App.tsx
const router = createBrowserRouter([
  {
    element: <RootProviders />,
    children: [
      { path: '/', element: <HomePage /> },
      { path: '/login', element: <PublicRoute><LoginPage /></PublicRoute> },
      { path: '/menu', element: <ProtectedRoute><MenuPage /></ProtectedRoute> },
      {
        element: <ProtectedRoute><SinglePlayerGameLayout /></ProtectedRoute>,
        children: [
          { path: '/game/single', element: <GamePage /> },
          { path: '/results', element: <ResultsPage /> },
        ],
      },
      // ... more routes
    ],
  },
]);
```

### Route Guards

- **ProtectedRoute**: Redirects to `/login` if user is not authenticated
- **PublicRoute**: Redirects to `/menu` if user is already logged in
- **AuthRouteLoading**: Shows loading state during auth check

### GitHub Pages Support

The app supports deployment to GitHub Pages with dynamic base path:

```ts
// vite.config.ts

```

Set `VITE_BASE_PATH=/geochallenge/` for deployment.

## State Management

### AuthContext

Manages authentication state globally:

```tsx
const { user, isLoading, login, logout, register } = useAuth();
```

**Features:**
- Token stored in localStorage
- Automatic session validation on mount
- Socket connection management
- Email/username normalization (v1.2.49)

### GameContext

Manages single-player game state:

```tsx
const { 
  questions, 
  currentQuestionIndex, 
  score, 
  submitAnswer, 
  nextQuestion 
} = useGame();
```

**Features:**
- Question loading and caching
- Answer validation
- Score calculation
- Game result submission

## Mobile-First Design

### Viewport Units

**IMPORTANT:** Use `dvh` (dynamic viewport height) instead of `vh` to avoid issues with mobile browser bars:

```css
.game-container {
  height: 100dvh; /* ✅ Correct */
  /* height: 100vh; ❌ Avoid */
}
```

### Safe Areas

Respect device notches and home indicators:

```css
.header {
  padding-top: env(safe-area-inset-top);
}

.footer {
  padding-bottom: env(safe-area-inset-bottom);
}
```

### Zero-Scroll Policy (Game Screens)

Game screens MUST NOT scroll. Use fixed layouts:

```tsx
<UniversalGameLayout>
  <header /> {/* Fixed top */}
  <main />   {/* Flexible content */}
  <footer /> {/* Fixed bottom */}
</UniversalGameLayout>
```

See `AGENTS.md` for full layout rules.

## Real-Time Features

Socket.IO is used for duels and challenges:

```tsx
// services/socket.ts
import { io, Socket } from 'socket.io-client';

const socket: Socket = io(API_URL, {
  auth: { token: localStorage.getItem('token') },
});

// Listen for events
socket.on('duel:matched', (data) => {
  console.log('Matched with opponent:', data.opponentName);
});

// Emit events
socket.emit('duel:join', { category: 'FLAG' });
```

### Connection Management

- Socket connects on login
- Disconnects on logout
- Auto-reconnects on network issues
- Keep-alive pings every 4 minutes (v1.2.14)

## Testing Strategy

### Unit Tests (Vitest)

Test components in isolation:

```tsx
import { render, screen } from '@testing-library/react';
import { Button } from './Button';

test('Button renders with text', () => {
  render(<Button>Click me</Button>);
  expect(screen.getByText('Click me')).toBeInTheDocument();
});
```

**Run:** `npm run test`

### E2E Tests (Playwright)

Test full user flows:

```tsx
import { test, expect } from '@playwright/test';

test('User can login', async ({ page }) => {
  await page.goto('/');
  await page.click('text=Iniciar sesión');
  await page.fill('input[type="email"]', 'test@example.com');
  await page.fill('input[type="password"]', 'password');
  await page.click('button[type="submit"]');
  await expect(page).toHaveURL('/menu');
});
```

**Run:** `npm run test:e2e`

### Mobile Testing

Playwright tests include mobile viewports:

```tsx
test.use({ viewport: { width: 375, height: 667 } }); // iPhone SE
test.use({ viewport: { width: 393, height: 852 } }); // iPhone 15 Pro
```

## Best Practices

1. **No mock data in production** - Always fetch real data from the API
2. **Mobile-first** - Design for small screens first, enhance for desktop
3. **Test everything** - All features must have unit or E2E tests
4. **Dark mode always** - Validate styles in both themes
5. **Validate layout** - Ensure no clipping or horizontal overflow
6. **Use design tokens** - No hardcoded colors (`#3B82F6` ❌ `--primary` ✅)
7. **Semantic HTML** - Use proper tags for accessibility
8. **TypeScript strict mode** - No `any` types without justification
